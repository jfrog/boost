Agent Workflows

Use the Boost skill

The Boost skill is installed into your coding agent by boost init and refreshed by boost update. In Cursor, it appears as the /boost command.

Use /boost to turn your local Boost history into concrete engineering work: flaky test candidates, slow CI commands, and noisy agent loops that waste time and tokens. The examples below show prompts you can run after Boost has been collecting traces in your project or CI.

Before you start

Install Boost, initialize your project, and restart Cursor so the generated /boost command is available. If Boost is already installed, run boost update to refresh the command and skill files. 

boost init
boost update
boost test ./...
boost npm test
boost gh run list --limit 10

Then let your agent or CI run a few real commands so there is trace data to analyze. If you have CI enabled, run boost init --github-actions so workflow commands are wrapped and traced as well.

Example 1

Detect flaky tests

Use this when a suite sometimes fails but reruns pass, or when CI has unexplained red builds. Ask the skill to compare recent test outcomes instead of guessing from one log.

Prompt

/boost flaky tests

Analyze test traces from the last 7 days.

Find flaky tests in this repo. For each candidate, show:
- test name and file
- pass/fail/skip counts
- failure rate
- the most common failure signature
- the first fix or quarantine step you recommend
Agent gpt-5.5

Response

Likely flaky tests

tests/e2e/checkout.spec.ts — checkout retries payment after timeout

31 passes, 5 failures, 0 skips. Failure rate: 13.9%. Common signature: timeout waiting for payment-confirmed.

Recommended next step: replace the fixed wait with a state poll, then rerun this test 20 times under Boost.

internal/cache/cache_test.go — TestEvictsExpiredEntries

48 passes, 3 failures, 0 skips. Failure rate: 5.9%. Common signature: expected item to expire within 10ms.

Recommended next step: inject a fake clock before changing production cache behavior.

Example 2

Find what slows CI

Use this when CI feels slow but it is not obvious whether the problem is install time, tests, Docker builds, cache misses, or a single expensive command.

Prompt

/boost slow CI

Inspect recent GitHub Actions traces.

Show the slowest CI steps by p90 duration for the last 20 runs. Separate:
- dependency install
- build
- test
- Docker
- deployment or publish steps

Call out cache miss patterns and recommend the highest-impact fix first.
Agent gpt-5.5

Response

CI bottlenecks by p90 duration

docker build . — p90 8m 12s

Layer cache hit rate dropped from 74% to 19% after the base image tag changed.

npm ci — p90 2m 44s

The package manager cache is restored, but node_modules is rebuilt every run.

go test ./... — p90 1m 31s

Most of the time is concentrated in integration tests that start containers serially.

Highest-impact fix: pin the Docker base image digest and restore BuildKit cache before the build step.

Example 3

Improve agent development loops

Use this when your agent keeps spending time on broad commands, huge logs, or repeated checks that could be narrowed to a faster validation path.

Prompt

/boost slow commands

Review my agent command history for this branch.

Find commands that slow down development or waste context. For each one:
- show how often it ran
- show total time and token savings
- explain why it was expensive
- suggest a narrower command the agent should run first next time
Agent gpt-5.5

Response

Agent loop improvements

go test ./... ran 14 times, 21m total

Expensive because the edited files only touched internal/handlers, but every package was tested.

Run first next time: boost go test ./internal/handlers/..., then broaden before final verification.

npm run build ran 6 times, 9m total

The failures were TypeScript errors in one route, so full production builds repeated work.

Run first next time: boost npm run astro check or the repo's type-check script.

git diff produced 84k raw output tokens

Boost compressed the output, but the agent still asked for the whole branch diff repeatedly.

Run first next time: boost git diff --stat, then inspect only changed files that matter.
